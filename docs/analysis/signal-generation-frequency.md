# Signal Generation Frequency Analysis

## Overview

This document analyzes how often trading signals are generated by the ML models in the JackSparrow Trading Agent system. Signals are generated through an event-driven architecture that responds to market data changes, specifically **major price fluctuations** (≥0.5% by default) rather than fixed time intervals.

## Signal Generation Flow

### Event Chain

The signal generation process follows this event-driven chain:

```
PriceFluctuationEvent → FeatureRequestEvent → FeatureComputedEvent →
ModelPredictionRequestEvent → ModelPredictionCompleteEvent →
ReasoningRequestEvent → DecisionReadyEvent → signal_update (WebSocket)
```

**Note**: The system also supports candle-based signals for longer-term analysis, but price fluctuation events are the primary trigger for ML pipeline execution.

### Detailed Flow

1. **Price Fluctuation Detection** (`agent/data/market_data_service.py`)
   - Market data service continuously monitors BTCUSD ticker prices (every 0.5 seconds)
   - When price changes by ≥0.5% from last major fluctuation, emits `PriceFluctuationEvent`
   - Tracks percentage change, volume, and timestamp of significant movements

2. **Feature Computation** (`agent/events/handlers/market_data_handler.py`)
   - Handles `PriceFluctuationEvent` (primary trigger) or `CandleClosedEvent` (secondary)
   - Triggers `FeatureRequestEvent` for all 50 required features
   - Features include: price-based (16), momentum (10), trend (8), volatility (8), volume (6), returns (2)

3. **Model Prediction** (`agent/events/handlers/feature_handler.py`)
   - After features are computed, triggers `ModelPredictionRequestEvent`
   - ML models generate predictions with confidence scores
   - Emits `ModelPredictionCompleteEvent` with consensus signal

4. **Reasoning Chain** (`agent/events/handlers/model_handler.py`)
   - Triggers `ReasoningRequestEvent` after model predictions complete
   - Reasoning engine generates 6-step reasoning chain
   - Synthesizes final decision from model consensus

5. **Decision Ready** (`agent/core/reasoning_engine.py`)
   - Emits `DecisionReadyEvent` with signal (BUY/SELL/HOLD), confidence, and reasoning
   - Includes timestamp from reasoning chain creation

6. **Frontend Broadcast** (`backend/services/agent_event_subscriber.py`)
   - Receives `DecisionReadyEvent` via Redis Streams
   - Formats as `signal_update` message
   - Broadcasts via WebSocket to all connected frontend clients

## Signal Frequency

### Primary Trigger: Price Fluctuations

Signals are generated when BTCUSD price experiences major fluctuations (≥0.5% change from the last major fluctuation). The frequency depends on:

- **Fluctuation Threshold**: `PRICE_FLUCTUATION_THRESHOLD_PCT` (default: `0.5%`)
- **Polling Frequency**: `FAST_POLL_INTERVAL` (default: `0.5` seconds)
- **Market Volatility**: More signals during volatile periods, fewer during stable periods

### Configuration

**Environment Variables**:
- `PRICE_FLUCTUATION_THRESHOLD_PCT`: Percentage threshold for triggering ML pipeline (default: `0.5`)
- `FAST_POLL_INTERVAL`: Continuous polling interval in seconds (default: `0.5`)
- `TIMEFRAMES`: Comma-separated list of timeframes for candle-based analysis (default: `15m,1h,4h`)
- `UPDATE_INTERVAL`: Legacy setting for candle-based operations (default: `900` = 15 minutes)

**Default Behavior**:
- Fluctuation threshold: `0.5%` (0.5% price change)
- Polling frequency: `0.5` seconds (continuous monitoring)
- Expected signal frequency: **Variable - depends on market volatility**
- Minimum frequency: As often as price moves ≥0.5%
- Maximum frequency: Limited by polling frequency and API rate limits

### Signal Frequency Examples

**High Volatility Market** (e.g., BTCUSD moving 2-5% per hour):
- Signals: ~4-10 per hour (every major price swing)
- More responsive to actual market movements
- Captures significant trend changes immediately

**Moderate Volatility Market** (e.g., BTCUSD moving 0.5-2% per hour):
- Signals: ~1-4 per hour
- Balanced responsiveness without excessive noise

**Low Volatility Market** (e.g., BTCUSD moving <0.5% per hour):
- Signals: ~0-1 per hour
- System remains alert but doesn't overreact to minor fluctuations

### FAST_POLL_INTERVAL Relationship

The `FAST_POLL_INTERVAL` setting (default: 0.5 seconds) controls:

1. **API Polling Frequency**: How often the system checks for price updates
2. **Responsiveness**: Lower values = more responsive but higher API usage
3. **Rate Limiting**: Prevents excessive API calls while maintaining real-time feel

The market data service uses:
```python
await asyncio.sleep(settings.fast_poll_interval)
```

This ensures continuous monitoring while respecting API rate limits.

## Edge Cases

### HOLD Signals

- **Frequency**: HOLD signals are generated at the same frequency as trading signals
- **Behavior**: When reasoning chain concludes with HOLD, `DecisionReadyEvent` is still emitted
- **Frontend Impact**: Frontend receives signal_update with `signal: "HOLD"`
- **State Machine**: Agent transitions from DELIBERATING → OBSERVING on HOLD

### No Significant Fluctuation

- **State Machine**: Agent remains in OBSERVING state
- **No Signal**: No `DecisionReadyEvent` is emitted if price change < fluctuation threshold
- **Frequency Impact**: During stable periods, signal frequency drops to zero
- **Monitoring**: System continues price monitoring but doesn't trigger ML pipeline

### Rapid Fluctuations

- **Debouncing**: Multiple small fluctuations may accumulate to trigger one signal
- **Processing Order**: Signals are processed sequentially through the event bus
- **Latest Signal Wins**: Frontend displays the most recent signal received
- **API Rate Limits**: Continuous polling respects rate limits to avoid being blocked

### Model Prediction Failures

- **Error Handling**: If model predictions fail, no `ModelPredictionCompleteEvent` is emitted
- **No Signal**: Reasoning chain is not triggered without model predictions
- **Fallback**: System continues monitoring but no signal is generated

## Signal Types

### Signal Values

- `STRONG_BUY`: High confidence buy signal (position size: 0.1)
- `BUY`: Moderate buy signal (position size: 0.05)
- `HOLD`: No trading action recommended
- `SELL`: Moderate sell signal (position size: 0.05)
- `STRONG_SELL`: High confidence sell signal (position size: 0.1)

### Signal Generation Logic

Signals are extracted from the reasoning chain conclusion:
```python
if "STRONG_BUY" in conclusion:
    signal = "STRONG_BUY"
elif "BUY" in conclusion:
    signal = "BUY"
elif "STRONG_SELL" in conclusion:
    signal = "STRONG_SELL"
elif "SELL" in conclusion:
    signal = "SELL"
else:
    signal = "HOLD"
```

## Real-World Frequency Expectations

### Typical Scenarios

1. **High Volatility Market** (BTCUSD moving 2-5% per hour):
   - Signals: ~4-10 per hour (responsive to major moves)
   - Captures significant trend changes immediately
   - More trading signals during volatile periods

2. **Moderate Volatility Market** (BTCUSD moving 0.5-2% per hour):
   - Signals: ~1-4 per hour
   - Balanced responsiveness without excessive noise
   - Good balance of timeliness vs. signal quality

3. **Low Volatility Market** (BTCUSD moving <0.5% per hour):
   - Signals: ~0-1 per hour (may go silent during extended stable periods)
   - System remains vigilant but avoids overtrading
   - Preserves capital during ranging/quiet markets

### Benefits of Fluctuation-Based Approach

- **Adaptive Frequency**: Signal generation automatically adjusts to market conditions
- **Real-Time Responsiveness**: Captures major moves immediately rather than waiting for time intervals
- **Reduced Noise**: Avoids unnecessary analysis during stable periods
- **Cost Efficiency**: Lower computational load during low-volatility periods

### Monitoring Recommendations

- Track signal generation rate relative to price volatility
- Monitor HOLD vs. trading signal ratio
- Alert if no signals received for >2 hours during moderate volatility
- Track signal age and market responsiveness
- Monitor API usage to ensure rate limits aren't exceeded
