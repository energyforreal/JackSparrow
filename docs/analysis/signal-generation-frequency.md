# Signal Generation Frequency Analysis

## Overview

This document analyzes how often trading signals are generated by the ML models in the JackSparrow Trading Agent system. Signals are generated through an event-driven architecture that responds to market data changes, specifically candle close events.

## Signal Generation Flow

### Event Chain

The signal generation process follows this event-driven chain:

```
CandleClosedEvent → FeatureRequestEvent → FeatureComputedEvent → 
ModelPredictionRequestEvent → ModelPredictionCompleteEvent → 
ReasoningRequestEvent → DecisionReadyEvent → signal_update (WebSocket)
```

### Detailed Flow

1. **Candle Close Detection** (`agent/data/market_data_service.py`)
   - Market data service monitors candle data
   - When a candle closes, emits `CandleClosedEvent`
   - Includes symbol, interval, OHLCV data, and timestamp

2. **Feature Computation** (`agent/events/handlers/market_data_handler.py`)
   - Handles `CandleClosedEvent`
   - Triggers `FeatureRequestEvent` for all 50 required features
   - Features include: price-based (16), momentum (10), trend (8), volatility (8), volume (6), returns (2)

3. **Model Prediction** (`agent/events/handlers/feature_handler.py`)
   - After features are computed, triggers `ModelPredictionRequestEvent`
   - ML models generate predictions with confidence scores
   - Emits `ModelPredictionCompleteEvent` with consensus signal

4. **Reasoning Chain** (`agent/events/handlers/model_handler.py`)
   - Triggers `ReasoningRequestEvent` after model predictions complete
   - Reasoning engine generates 6-step reasoning chain
   - Synthesizes final decision from model consensus

5. **Decision Ready** (`agent/core/reasoning_engine.py`)
   - Emits `DecisionReadyEvent` with signal (BUY/SELL/HOLD), confidence, and reasoning
   - Includes timestamp from reasoning chain creation

6. **Frontend Broadcast** (`backend/services/agent_event_subscriber.py`)
   - Receives `DecisionReadyEvent` via Redis Streams
   - Formats as `signal_update` message
   - Broadcasts via WebSocket to all connected frontend clients

## Signal Frequency

### Primary Trigger: Candle Closes

Signals are generated when candles close on configured timeframes. The frequency depends on:

- **Primary Timeframe**: First timeframe in `TIMEFRAMES` configuration (default: `15m`)
- **Configured Timeframes**: Comma-separated list (default: `15m,1h,4h`)
- **Candle Close Events**: One signal per candle close on primary timeframe

### Configuration

**Environment Variables**:
- `TIMEFRAMES`: Comma-separated list of timeframes (default: `15m,1h,4h`)
- `UPDATE_INTERVAL`: Fallback polling interval in seconds (default: `900` = 15 minutes)
- `AGENT_INTERVAL`: Primary interval if timeframes not specified (default: `15m`)

**Default Behavior**:
- Primary timeframe: `15m` (15 minutes)
- Expected signal frequency: **One signal every 15 minutes** (when candle closes)
- Minimum frequency: Every 15 minutes
- Maximum frequency: Could be more frequent if multiple timeframes trigger simultaneously

### Timeframe to Seconds Mapping

| Timeframe | Seconds | Signals per Hour | Signals per Day |
|-----------|---------|------------------|-----------------|
| 1m        | 60      | 60               | 1,440           |
| 5m        | 300     | 12               | 288             |
| 15m       | 900     | 4                | 96              |
| 30m       | 1,800   | 2                | 48              |
| 1h        | 3,600   | 1                | 24              |
| 4h        | 14,400  | 0.25             | 6               |
| 1d        | 86,400  | 0.042            | 1               |

**Note**: With default `15m` timeframe, signals are generated **4 times per hour** or **96 times per day**.

### UPDATE_INTERVAL Relationship

The `UPDATE_INTERVAL` setting (default: 900 seconds = 15 minutes) serves as:

1. **Fallback Polling Frequency**: Used by market data service when no candle close events occur
2. **Minimum Sleep Duration**: Ensures market data service doesn't poll too frequently
3. **Not Direct Signal Trigger**: Signals are primarily triggered by candle closes, not by UPDATE_INTERVAL

The market data service uses:
```python
sleep_seconds = max(1, min(update_interval, interval_seconds))
```

This ensures polling respects both the UPDATE_INTERVAL and the candle interval.

## Edge Cases

### HOLD Signals

- **Frequency**: HOLD signals are generated at the same frequency as trading signals
- **Behavior**: When reasoning chain concludes with HOLD, `DecisionReadyEvent` is still emitted
- **Frontend Impact**: Frontend receives signal_update with `signal: "HOLD"`
- **State Machine**: Agent transitions from DELIBERATING → OBSERVING on HOLD

### No Significant Change

- **State Machine**: Agent remains in OBSERVING state
- **No Signal**: No `DecisionReadyEvent` is emitted if no significant market change is detected
- **Frequency Impact**: This can reduce actual signal frequency below theoretical maximum

### Multiple Timeframes

- **Simultaneous Triggers**: If multiple timeframes close simultaneously, multiple signals may be generated
- **Processing Order**: Signals are processed sequentially through the event bus
- **Latest Signal Wins**: Frontend displays the most recent signal received

### Model Prediction Failures

- **Error Handling**: If model predictions fail, no `ModelPredictionCompleteEvent` is emitted
- **No Signal**: Reasoning chain is not triggered without model predictions
- **Fallback**: System continues monitoring but no signal is generated

## Signal Types

### Signal Values

- `STRONG_BUY`: High confidence buy signal (position size: 0.1)
- `BUY`: Moderate buy signal (position size: 0.05)
- `HOLD`: No trading action recommended
- `SELL`: Moderate sell signal (position size: 0.05)
- `STRONG_SELL`: High confidence sell signal (position size: 0.1)

### Signal Generation Logic

Signals are extracted from the reasoning chain conclusion:
```python
if "STRONG_BUY" in conclusion:
    signal = "STRONG_BUY"
elif "BUY" in conclusion:
    signal = "BUY"
elif "STRONG_SELL" in conclusion:
    signal = "STRONG_SELL"
elif "SELL" in conclusion:
    signal = "SELL"
else:
    signal = "HOLD"
```

## Real-World Frequency Expectations

### Typical Scenarios

1. **Active Market (15m timeframe)**:
   - Signals: ~4 per hour, ~96 per day
   - Most signals will be HOLD during normal market conditions
   - Trading signals (BUY/SELL) occur less frequently

2. **Low Volatility**:
   - More HOLD signals
   - Fewer trading signals
   - Frequency remains consistent (candle closes still trigger analysis)

3. **High Volatility**:
   - More trading signals
   - More frequent state transitions
   - Same base frequency (15 minutes)

### Monitoring Recommendations

- Track signal generation rate vs. theoretical maximum
- Monitor HOLD vs. trading signal ratio
- Alert if no signals received for >30 minutes (2 candle periods)
- Track signal age to ensure freshness
